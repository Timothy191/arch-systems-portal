Single-file stdio-based MCP server (`index.js`) built on `@modelcontextprotocol/sdk` v1.8.0; it registers eight tools (`n8n_list_workflows`, `n8n_get_workflow`, `n8n_execute_workflow`, `n8n_check_execution`, `n8n_import_workflow`, `n8n_activate_workflow`, `n8n_deactivate_workflow`, `n8n_search_workflows_by_tag`) via `ListToolsRequestSchema`/`CallToolRequestSchema` handlers and talks to a remote n8n instance over HTTP using cookie-based session auth (login via `/rest/login`, then re-auth on 401). All outbound calls go through `n8nFetch`, which centralizes timeout handling, cookie injection, and retry-on-401 logic. Input parameters are validated with a shared `sanitizePathSegment` helper before being interpolated into REST paths. The runtime is configured as an ES module (`"type": "module"`) and launched via `StdioServerTransport`. Deployment is managed by `project.json` (Nx tag `scope:tool`) and `n8n.service` (systemd unit). The `workflows/` directory holds declarative n8n JSON definitions — ten top-level orchestration workflows (batcher, function orchestrator, skills loader, vector/hybrid memory, orchestrator-worker, evaluator-optimizer, parallel-executor, observability-pipeline, guardrails) plus five automation-mesh sub-workflows — imported into a live n8n instance by `import-workflows.sh`, which logs in once and POSTs each file to `/rest/workflows` then activates any inactive ones.
