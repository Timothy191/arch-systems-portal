The workspace is split into two independent package ecosystems glued together at the root:

- Python side: `packages/core`, `packages/cli`, and `packages/server` are mapped to namespace packages under the `repowise.*` top-level via `[tool.setuptools.package-dir]` in `pyproject.toml`; they are published as one wheel (`repowise`) with console scripts `repowise`, `repowise-augment`, `repowise-rewrite`. `core` owns ingestion, graph building, dead-code detection, health scoring, distillation, and persistence; `cli` (Click) and `server` (FastAPI + MCP) are thin shells that import from `repowise.core` and register commands/tools through the shared `repowise.core.registry` so third-party plugins can extend both entry points without monkey-patching.
- TypeScript side: `packages/types`, `packages/ui`, `packages/api-client`, `packages/web`, `packages/vscode` form a pnpm/npm workspaces monorepo driven by the root `package.json`; `@repo/ui` is the shared component layer consumed by `web` and `vscode`, while `api-client` provides the typed HTTP client used by both.
- Cross-cutting wiring: `scripts_and_docker` bootstraps Supabase + Next.js for local dev and ships systemd/Docker services that run the same `repowise` wheel installed from PyPI; `tests/` mirrors the Python package layout (unit/integration/e2e) and validates the core+CLI+server contract end-to-end.
