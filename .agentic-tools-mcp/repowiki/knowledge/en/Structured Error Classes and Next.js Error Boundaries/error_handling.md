The portal app implements a layered error-handling strategy centered on typed error classes, centralized logging, and Next.js App Router error boundaries.

**Error class hierarchy** — `apps/portal/lib/errors/error-classes.ts` defines an `AppError` base with `code`, `statusCode`, `context`, and `cause` fields, plus domain subclasses: `APIError`, `AuthError`, `DatabaseError`, `ValidationError`, `NotFoundError`, `ConflictError`, `ForbiddenError`. Each subclass normalizes defaults (e.g. `AuthError` → 401, `ValidationError` → 400, `NotFoundError` → 404). A shared `resolveOptions` helper accepts either `(message, code?, statusCode?)` or `(message, { code, statusCode, context, cause, ...extra })`, merging extra keys into `context` so call sites stay concise. Type guards (`isAppError`, `isAuthError`, `isValidationError`, `isNotFoundError`) let UI layers branch on error shape.

**Logging pipeline** — `apps/portal/lib/errors/error-logger.ts` provides `logError(error, ctx)` which builds a structured `ErrorLogEntry` (timestamp, severity derived from `statusCode`, code, message, context, stack, url, method, userId, sessionId) and forwards it to the console in development and to Sentry for `error`/`fatal` severities only. Two wrappers exist: `withErrorLogging(req, handler)` for Route Handlers and `withServerActionLogging(handler, actionName)` for server actions; both re-throw after logging so they can be used as thin try/catch shims without swallowing errors.

**Next.js error boundaries** — Three boundary files render user-facing pages:

- `app/global-error.tsx`: top-level `<html>` shell shown when React cannot mount any component; displays a "System Error" card with the error digest and a reload button.
- `app/error.tsx`: per-route fallback that inspects the error via the type guards, maps `NotFoundError` → "Page not found", `AuthError` → "Access denied", `ValidationError` → "Invalid input", otherwise falls back to the `AppError.name` / message; shows the error code and numeric status, and in development renders the full `context` object inside a collapsible block. It calls `logError` on mount so every caught error is captured.
- `app/not-found.tsx`: dedicated 404 page rendered by Next.js when no route matches.
- Per-group boundaries under `app/(departments)/[department]/error.tsx` and `app/(hub)/error.tsx` specialize titles like "Department not found" / "Hub not found".

**Middleware authorization errors** — `apps/portal/middleware.ts` does not throw; instead it redirects with query-string error codes via `redirectWithError(request, 'unauthorized_department'|'unknown_department', response)`, preserving cookies across the redirect. Token-expired Supabase responses are detected through `isTokenExpiredError` and trigger an automatic sign-out + cache eviction.

**Propagation conventions** — Business logic throws typed `AppError` subclasses carrying machine-readable `code` strings and developer-friendly `message`s; API routes wrap handlers in `withErrorLogging` so unhandled exceptions bubble to the nearest error boundary rather than crashing the process. The logger itself is wrapped in a try/catch so monitoring failures never propagate back to application code.
