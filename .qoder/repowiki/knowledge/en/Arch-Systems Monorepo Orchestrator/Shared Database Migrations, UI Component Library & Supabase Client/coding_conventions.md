- Migration files are strictly ordered with zero-padded numeric prefixes (e.g. `001_initial.sql` … `067_cache_events.sql`) and each file must be idempotent — they are applied via `supabase db push` and also used as seed during `db reset`.
- Public components in `@repo/ui` are exposed through explicit `package.json` `exports` mappings keyed by stable names (e.g. `"./CyberButton": "./src/components/ui/cyber-button.tsx"`) rather than relying on directory traversal imports.
- Supabase client construction follows a per-context pattern: one factory per execution environment (`createBrowserSupabaseClient`, `createMiddlewareClient`, `createServerSupabaseClient`) each wrapping `@supabase/ssr`'s `createBrowserClient`/`createServerClient` with cookie handling appropriate to that layer.
- Environment-driven URL rewriting is centralized in the browser client factory: when running on localhost the configured `NEXT_PUBLIC_SUPABASE_URL` hostname is rewritten to `127.0.0.1` so the client talks to the local Supabase instance automatically.
