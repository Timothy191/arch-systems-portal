Flat Next.js App Router layout under `apps/portal/app/api/export/<resource>/route.ts`, one GET handler per exportable domain. Each route follows the same internal flow: create a Supabase server client via `@repo/supabase/server`, authenticate with `supabase.auth.getUser()`, parse query params through a shared Zod schema (`exportQuerySchema` or `safetyExportQuerySchema` from `@/lib/api/schemas`), build a paginated Supabase query on the underlying table (`daily_logs`, `machines`, `safety_incidents`) with optional `dept`-to-id lookup against `departments`, then branch on `Accept: text/csv` to either stream a `sanitizeCsvCell`-escaped CSV attachment or return a JSON envelope carrying `data`, `count`, `limit`, `offset` (and `from`/`to` date bounds). All responses are wrapped in `applyCors` and gated by `withRateLimit`. The module has no internal sub-packages — it is purely a collection of parallel handlers sharing cross-cutting concerns (auth, rate limiting, CORS, param validation) from `@/lib/api/*`.
