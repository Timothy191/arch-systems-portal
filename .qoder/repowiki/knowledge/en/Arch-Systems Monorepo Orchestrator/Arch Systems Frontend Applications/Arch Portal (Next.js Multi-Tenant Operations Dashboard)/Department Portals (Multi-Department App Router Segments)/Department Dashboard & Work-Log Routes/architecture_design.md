- Route layout: a shared `layout.tsx` resolves the department slug via `DEPARTMENTS`/`getDepartmentTabs`, renders it inside `@repo/ui/DepartmentLayout`, injects an `ActiveDepartmentSetter` and an `AIAssistantWrapper`, and calls `notFound()` for unknown slugs. Each sub-folder is a Next.js route (`page.tsx`) with optional `loading.tsx` (RSC Suspense fallback) and `error.tsx`.
- Data fetching pattern: every page uses `getDepartmentContext({ department })` from `~/lib/dept-context` to obtain `deptId`, `supabase`, and `today`; queries are issued against Supabase tables (`machine_operations`, `daily_logs`, `operational_delays`, `hourly_loads`, `machines`, `engineering_notes`, `breakdowns`, `safety_incidents`, …). Control-room dashboards additionally wrap heavy widgets in `Suspense` + `dynamic()` imports with skeleton placeholders.
- Server actions: mutation endpoints live alongside their route as `actions.ts` files (e.g. `hourly-loads/actions.ts`) marked with `'use server'`. They re-authenticate via `createServerSupabaseClient`, enforce RLS-equivalent checks (employee role + department_id), then escalate to `createServiceRoleClient` only for admin-only writes like updating `machines.site_id`.
- Department scoping: some pages gate access with `requireDepartment(department, "control-room")` (Engineering Notes); others branch on `dept.type === 'satellite' | 'safety' | 'control_room'` to swap entire dashboards.
- Error boundary: a single client-side `error.tsx` dispatches on custom error classes (`isNotFoundError`, `isAuthError`, `isAppError`) to pick title/message/action and always forwards to `logError`.
