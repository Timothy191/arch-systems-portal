Two-file leaf module with no internal sub-packages:
- `error-classes.ts` declares a local replacement for the shared `@repo/errors` package (noted in the file header). It exposes an `AppError` base class plus domain-specific subclasses (`APIError`, `AuthError`, `DatabaseError`, `ValidationError`, `NotFoundError`, `ConflictError`, `ForbiddenError`) that each normalize a flexible constructor signature — `(message, code?, statusCode?)` or `(message, options)` — through a shared `resolveOptions` helper. Extra keys on the options bag are merged into `context`. Type guards (`isAppError`, `isValidationError`, etc.) are exported alongside.
- `error-logger.ts` provides the logging surface: `logError(error, context)` builds an `ErrorLogEntry` (timestamp, severity derived from status code, code/status/context/cause), writes to console, and forwards error/fatal entries to `@sentry/nextjs` via `Sentry.captureException`. Two higher-order helpers wrap callers: `withErrorLogging(req, handler)` for Next.js API routes and `withServerActionLogging(handler, actionName)` for server actions; both re-throw after logging so existing error boundaries stay intact.
Dependency direction is one-way: the logger depends on the error classes only by duck-typing (`code`/`statusCode`/`context`/`cause` properties) rather than importing them, keeping the two files decoupled.
