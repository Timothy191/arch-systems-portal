Two-layer design built on XState v5 actors:
- `machines/` holds the state machines. `orchestrator.machine.ts` defines a top-level machine (`idle → initializing → loadingPlugins → active`) that spawns one child `pluginMachine` per configured plugin name (default: `predictive-maintenance`, `rust-telemetry-engine`, `buggy-plugin`). The orchestrator's context is a `Map<string, PluginActor>` plus a derived `HealthReport`; it forwards `RETRY_PLUGIN` / `ENABLE_PLUGIN` / `DISABLE_PLUGIN` events to the matching child.
- `plugin.machine.ts` models a single plugin's lifecycle (`idle → loading → validating → active | disabled | failed → retrying → loading`) with `fromPromise`-based async load, guarded retries (`canRetry`, `isRetryableError`), and a `loadPluginModule` helper that does a dynamic `import('../../plugins/${pluginName}/index')` against the host app's `apps/portal/lib/plugins/<name>/index` layout.
- `machines/types.ts` centralises shared event/context types and the `isRetryableError` matcher; `machines/index.ts` re-exports only `orchestratorMachine` + `HealthReport`.
- `orchestrator.ts` wraps the actor in a synchronous-looking `PluginOrchestrator` class (singleton `pluginOrchestrator`) exposing `loadAllPlugins`, `executeEngine`, `triggerHook`, `getActiveWidgets`, `retryPlugin`, `enablePlugin`, `disablePlugin`, `subscribe`. It guards server-only execution with `typeof window !== 'undefined'`, starts the actor on Node, polls `LOAD_PLUGINS` completion with a 10s timeout, and routes hook invocations through `Promise.allSettled` so one failing hook never blocks others.
- `types.ts` declares the public plugin contract (`ArchPlugin` = `metadata` + optional `engine` + `hooks` + `widgets` + `workflow`); this is the only cross-boundary type between the orchestrator and actual plugin implementations.
Dependency direction: `orchestrator.ts` → `machines/*` → `types.ts`; plugin implementations import only `types.ts` and are loaded at runtime via dynamic import — no compile-time coupling.
