Two distinct sub-scopes share the `@/lib/api` namespace:

- `apps/portal/lib/api/` — framework-agnostic middleware primitives consumed by Route Handlers. `response.ts` exports `validateBody`, which wraps a Zod schema against `request.json()` and returns either `{ data }` or a 400 `NextResponse`; `schemas.ts` is the single source of truth for every POST/PUT payload shape (webhooks, export, scanner/c66, telemetry push, sync playback, AI chat, admin data). `rate-limit-middleware.ts` implements a pluggable store (`MemoryStore` / `RedisStore`) behind a strategy interface (`TokenBucketStrategy` for `/api/ai/**`, `SlidingWindowStrategy` otherwise), with per-path limits resolved via `rate-limit-config.ts` and load-adaptive halving when `os.loadavg() > 0.85 * cpus`. `body-limit.ts` and `cors.ts` are thin wrappers returning early on violation.
- Loose files `weather-api.ts` and `monitoring-api.ts` are pure client modules: they call free third-party REST APIs (Open-Meteo, EOX WMTS, Copernicus STAC) using native `fetch` with Next.js ISR `revalidate` options, map raw responses into strongly-typed interfaces (`WeatherData`, `STACItem`, `DeformationReading`), and expose helper functions (`getWeatherDescription`, `classifyDeformationByVelocity`, `generateDeformationReadings`).

Dependency direction is one-way: route handlers → these helpers; the external-service clients have no dependency on the middleware layer.
