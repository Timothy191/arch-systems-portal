The `memex/` Python package is the single installable wheel (`memex-mcp`) whose top-level subpackages form one runtime: `graph/` owns the shared Neo4j schema, Graphiti client singleton and telemetry; every other subpackage imports from it. The `watcher/` daemon indexes filesystem/git events into that graph, while `mcp_server/` and `memory_tool/` read (and write) the same graph over stdio HTTP and FastAPI respectively. `cli_and_entrypoints` (`memex/cli.py`, `pyproject.scripts.memex`) is the process boundary that parses args and spawns either the watcher or the MCP/memory server, so all processes share configuration via `config.yaml` + env vars (`NEO4J_URI`, `GEMINI_API_KEY`, …). The sibling `memex-vscode/` TypeScript extension is the only non-Python child: it launches the local memex server as a subprocess and talks to its HTTP/SSE endpoints, rendering the graph in a webview. A separate `npm/` package wraps the same Python entry point for consumption by MCP host clients declared in `server.json`. Tests under `tests/` exercise each subpackage against stubbed or real Neo4j instances and are driven by pytest with `asyncio_mode=auto`.
